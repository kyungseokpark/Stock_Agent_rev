### 핵심 발견 5가지

1. **VCP 하드게이트(`min_vcp_score: 70`)가 최대 병목.** 게이트 제거 시 거래 22→60건, 기대R −0.03→+0.12, OOS 기대R −0.28→**+0.24**(유일한 OOS 양수), 총수익 −6.8%→+24.2%. 게이트를 통과한 VCP≥70 거래는 이 구간에서 5건뿐이라 통계적 우위 근거도 없음.
2. **목표가–보유기간 기하 불일치.** 기본 설정(2R 목표 = ATR×4 이동, 5일 보유)에서 목표 도달이 22건 중 **1건(4.5%)**, 시간청산 73%. 목표가는 사실상 장식. 보유 10일 확대만으로 기대R −0.03→+0.11. (WO-007 전체기간 top10/10d 결과와도 일관)
3. **레짐 게이트가 이 구간에서 0회 발동** — noregime과 결과 완전 동일. 상승장에서는 변별력이 없고, WO-007에서 확인했듯 하락장 조기감지가 실질 가치. 즉 레짐은 "미국장 상승기 성과"가 아니라 "국장형 급락 방어"로 평가해야 함.
4. **자본 가동률이 낮아 벤치마크 대비 구조적 열위.** 22건×5일 ≈ 110 포지션-일 / (3슬롯×131일=393 슬롯-일) ≈ **28% 가동률**. 나머지는 현금 유휴. 전략 대 벤치 비교가 노출 미조정 상태라 불공정하기도 함(→ F-2).
5. **점수 구간별 기대R 단조성 부족.** vcp0 기준 55–65점 +0.05, 65–75점 +0.18, 75+점 +0.12 — 고득점=고수익이 성립하지 않음. 스코어 캘리브레이션(F-7) 필요 근거.

## 2. 수정해야 할 로직 (P0 — 이 순서로)

### L-1. 백테스트 ↔ 실전 스크리너 선정 로직 불일치 해소 (최우선)
- **As-Is**: 실전(`main.run_screen`)은 VCP·RS를 **soft 필터**(통과자 없으면 완화)로 쓰고 `adjusted_score`/`composite_score`(세력유입 반영)로 정렬하지만, 백테스트(`run_picks_backtest`)는 `final_score` + **hard VCP/RS 게이트**로 전혀 다른 후보를 뽑는다. 지금 백테스트는 "실제 프로그램이 추천하는 종목"을 검증하지 않는다.
- **To-Be**: 후보 선정 블록을 공용 함수로 추출해 실전·백테스트가 동일 규칙을 쓰도록 통일. 최소한 (1) VCP를 hard 게이트에서 우선순위/보너스로 강등(`min_vcp_score: 0` 기본값 + 정렬 가점), (2) `scoring_adjustment`(chart_type 보너스)를 백테스트 점수에도 적용.
- **판정**: 동일 날짜에 대해 실전 스크리너 top5와 백테스트 그 날짜 후보 top5가 일치하는 회귀 테스트 추가.

### L-2. 청산 기하 재설계
- **As-Is**: stop=ATR×2.0, target=2R(=ATR×4), 보유 5일 → 목표 도달 4.5%.
- **To-Be**: 기본 `max_holding_days: 10`으로 변경(백테스트 근거 확보됨). target_r 상향(3.0)은 개선 없음(−0.02) → 기각. 부분청산·트레일링은 F-3에서 옵션으로.
- **판정**: h10 기준 기대R ≥ +0.10, PF ≥ 1.3 유지.

### L-3. 데이터 기간·캐시 정합
- **As-Is**: (1) 캐시가 period="1y"로 저장되어 백테스트 lookback "2y" 요청 시 `read_ohlcv_cache`의 **period 문자열 완전일치** 조건 때문에 캐시 전체 미스 → 매번 재다운로드. (2) `--offline`은 1y 캐시를 그대로 쓰는데 min_rows=120 소모 후 유효 재생이 131일뿐.
- **To-Be**: (1) 캐시 신선도 비교를 "요청 기간 ⊆ 보유 기간"으로 완화(2y 보유 캐시로 1y 요청 충족). (2) 기본 수집 period를 2y로 확대해 백테스트 표본 확보. (3) 백테스트 리포트에 데이터 기준일·유효 재생 일수 자동 표기.

### L-4. 동시터치(ambiguous) 집계
- **As-Is**: `_simulate_exit`는 같은 봉에서 stop·target 동시 터치 시 stop 우선(보수적)이나 집계가 없고, `performance_tracker`는 누적 `hit_stop`(80.6%)과 `hit_target1`(61.3%)을 따로 보여줘 합계 100% 초과로 오독을 유발.
- **To-Be**: 백테스트 trades에 `ambiguous` 플래그 추가·요약에 비율 표기. 성과 요약 텍스트는 `first_hit` 기준(최초 도달)으로 통일하고 누적 터치율은 보조 지표로 라벨 명확화.

### L-5. RSI 계산 방식
- **As-Is**: `calculate_rsi`가 단순 rolling mean — Wilder 평활(EMA α=1/14)이 아님. 표준 RSI 대비 과민하게 움직여 45–65 구간 점수·과열 페널티 판정이 어긋날 수 있음.
- **To-Be**: `ewm(alpha=1/period)` Wilder 방식으로 교체 후 점수 회귀 확인(임계값 재보정 필요 여부 포함). 교체가 부담이면 최소한 명칭/문서에 방식 명시.

### L-6. 비용 모델 정밀화
- **As-Is**: 왕복 비용을 **진입가 기준**으로만 계산(청산가 무시), KR 매도세 0.18% 하드코딩.
- **To-Be**: 진입·청산 각각의 체결가 기준으로 비용 계산. KR 거래세율은 현행 세율 확인 후 config 주석에 근거 명시(연도별 변동 있음 — 하드코딩 값 검증 필요).

### L-7. 레짐 프록시 왜곡
- **As-Is**: `classify_regime`에서 이력 200일 미만이면 `ma200 = 전체 평균`으로 대체 → 1y 캐시 환경에서 상시 왜곡.
- **To-Be**: 200일 미만 시 ma200 조건을 판정에서 제외(단기 조건만 사용)하거나 neutral 고정. L-3(2y 수집)과 병행 시 자연 해소.

### L-8. `parameter_sensitivity` 정합
- **As-Is**: 단일종목 이벤트 백테스트(`run_backtest`) 기반이라 실제 픽 재생과 무관한 민감도를 산출.
- **To-Be**: `run_picks_backtest` 기반 그리드(min_score × holding_days × target_r)로 교체, 캐시 재사용으로 속도 확보.

## 4. 프로그램 전반 문제점 (백테스트 외)

1. **실측 성과 경고등.** `output/kr/performance_summary.txt`: 완료 추천 5일 평균 **−8.88%**, 손절 터치 **94.7%** — WO-007 방어 패치 이후에도 국장 매수콜의 실측 성과 확인 필요. US도 목표 61%/손절 81% 동시 터치로 지표 자체가 모호(→ L-4).
2. **`desktop_app.py` 2,140줄 모놀리스.** UI·실행 오케스트레이션·설정이 한 파일. 유지보수/테스트 곤란 — 화면(tab)별 모듈 분리 권장.
3. **죽은 방어코드.** `signal_engine.py`의 Streamlit hot-reload TypeError 방어는 데스크톱 전환 후 잔재 — 제거 가능.
4. **스코어링 가중 중복.** `cp<ma20`이 trend 가점 상실(−10)과 penalty(−12)로 이중 반영되는 등 항목 간 중복이 많아 점수 해석성이 낮음. stop=`max(ma20, cp−ATR×k)` 구조는 cp가 ma20 바로 위일 때 RR이 폭주(120 등)해 cap+penalty로 땜질된 상태(WO-007) — stop 산정 자체 재검토가 근본 해법.
5. **미사용·불일치 코드.** `portfolio.calculate_position_size`(ATR 리스크 사이징)는 픽 재생 백테스트에서 미사용(슬롯 균등배분) — 실전 안내와 백테스트 손익 기준이 다름. `force_inflow`/`composite_score`도 백테스트에 미반영(→ L-1과 함께 정리).
6. **인코딩 깨짐.** `main.py` L214 `stats["force_real_fetch_mode"] = f"?? {real_fetch_limit}? ..."` — 한글이 `?`로 저장돼 있음. 원문 복구 필요.
7. **빌드 산출물 repo 포함.** `DAD/`(exe + _internal 전체)가 저장소에 포함 — 용량·혼란. 배포 폴더는 .gitignore 처리 권장.
8. **테스트 실행 환경.** `requirements.txt`에 pytest 명시 및 실행 문서화(이번 환경에는 pytest 미설치로 tests/ 실행 불가했음. `.pytest_cache/lastfailed`는 빈 상태 = 직전 실행 전부 통과).
9. **`--offline` 데이터 기준일 미표기.** 오프라인 재생 결과에 캐시 기준일이 없어 최신 결과로 오독 가능(→ L-3(3)).

## 5. 판정 기준 (Acceptance)

L-1~L-3 반영 후 오프라인 A/B 재실행(동일 캐시)에서:
- 거래수 ≥ 50 (표본 확보), **OOS 기대R > 0 유지**
- 기대R ≥ +0.10, PF ≥ 1.3, 승률 비열화
- MDD가 base(−17.6%) 대비 악화되지 않을 것
- 노출조정 수익률(F-2) 기준 벤치마크 열위 미전환
- L-1 회귀: 실전 스크리너 top5 = 백테스트 동일날짜 후보 top5 일치
